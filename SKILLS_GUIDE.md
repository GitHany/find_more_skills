# Hany 技能调用指南

## 技能调用标准格式

```
帮我 [具体任务描述]
```

## 1. find-skills - 发现技能

### 使用场景

- 不知道该用什么工具/技能
- 想找现成的解决方案
- 需要扩展 AI 能力

### 标准描述

```
帮我搜索 [关键词] 相关的技能
```

### 描述优化

**find-skills - 模糊搜索**

```
利用 find-skills 技能帮我搜索 React 状态管理和性能优化相关的技能。
我希望找到能够处理全局状态管理、组件性能监控、以及渲染优化的工具或技能库。
预期输出应包含具体的技能名称、适用场景和推荐理由。
```

**find-skills - 精准搜索**

```
利用 find-skills 技能帮我搜索前端 E2E 测试相关的技能，要求使用 Playwright 或 Cypress 框架。
需要覆盖端到端测试、跨浏览器兼容性测试、以及自动化测试脚本编写的技能。
输出应包含技能特点对比和适用场景分析。
```

**find-skills - 多关键词搜索**

```
利用 find-skills 技能帮我搜索 Docker 容器化和 Kubernetes 部署相关的技能。
我需要从容器化基础镜像构建、多容器编排、到 K8s 集群管理及 CI/CD 集成的完整技能链。
输出应包含技能间的关联关系和学习路径建议。
```

**find-skills - 带偏好搜索**

```
利用 find-skills 技能帮我搜索代码审查相关的技能，要求具备 AI 辅助功能。
我希望技能能够自动识别代码异味、生成审查建议、并支持自定义规则配置。
输出应包含 AI 功能的详细介绍和使用门槛说明。
```

### 完整示例

```
帮我搜索 React 项目性能优化相关的技能
```

## 2. hany-deep - 深度需求访谈

### 使用场景

- 想法模糊不确定
- 担心遗漏重要需求
- 不想被 AI 随意假设
- 复杂项目需要清晰规格

### 标准描述

```
帮我做一个需求访谈，确保理解 [我的想法/项目]
```

### 描述优化

**hany-deep - 全新项目**

```
使用 hany-deep 技能帮我做一个博客系统的需求访谈。
我的目标是创建一个能发表文章吸引读者的博客系统。
系统需要支持文章发布、分类标签、评论互动、SEO 优化等核心功能。
请在访谈中帮我明确目标用户画像、内容策略、以及与现有博客平台（如 WordPress、Hexo）的差异化方向。
```

**hany-deep - 复杂功能**

```
利用 hany-deep 技能帮我做一个用户认证系统的需求访谈。
系统需要包含用户注册登录、邮箱验证、密码找回、两步验证、OAuth 第三方登录、以及基于角色的权限管理（RBAC）。
我需要你帮我梳理用户流程、安全要求、 session 管理策略、以及与企业现有系统的集成方式。
```

**hany-deep - 强调澄清**

```
利用 hany-deep 技能帮我做深度访谈，确保完全理解我想要的电商系统。
核心需求包括商品管理、购物车、订单处理、支付集成、库存管理、物流跟踪等模块。
请特别注意帮我区分核心功能与扩展功能，明确 MVP 范围，不要自己假设任何需求，所有假设都需要我确认。
```

**hany-deep - 带约束**

```
利用 hany-deep 技能帮我做需求访谈。
我有明确的约束条件：预算限制在 [X] 万元以内，上线时间要求是 [具体日期]，技术团队有 [X] 人。
请帮我基于这些约束理清需求优先级，识别关键路径和非关键路径，提出分阶段交付的建议方案。
```

### 完整示例

```
帮我做一个 SaaS 产品后台管理系统的需求访谈
这个系统需要包含用户管理、数据看板、订阅计费等功能
我对技术栈没有特别要求，但希望系统易于扩展
请确保在开发前把所有需求都理清楚
```

## 3. hany-plus - 循环深度访谈

### 使用场景

- 有多个不相关的想法需要探索
- 需要批量做需求收集
- 想一次性规划多个模块

### 标准描述

```
帮我批量访谈多个需求：[需求1]、[需求2]、[需求3]
```

### 描述优化

**hany-plus - 多个模块**

```
使用 hany-plus 技能帮我批量访谈多个模块：用户认证、数据分析、报表导出。
针对每个模块，我需要明确功能范围、用户角色、数据来源、输出形式、以及与整体系统的集成关系。
请按优先级逐个进行深度访谈，每个模块都要达到可执行的规格说明水平。
```

**hany-plus - 探索方向**

```
利用 hany-plus 技能帮我系统性地探索包含社交功能、协作功能和自动化工作流在内的多个产品方向。
针对每个功能方向，需进行以下工作：
1. 市场需求分析：调研目标用户群体的具体需求、痛点及现有解决方案的不足之处
2. 功能设计规划：提出差异化的功能模块设计，明确核心功能与辅助功能的优先级
3. 技术可行性评估：结合当前技术栈，分析各功能方向的技术实现难度与资源需求
4. 商业模式构思：探索各产品方向可能的盈利模式与市场定位
5. 竞争优势分析：识别并提炼每个方向的独特竞争优势与壁垒
最终形成一份包含三个功能方向的详细产品探索报告。
```

**hany-plus - 完整规划**

```
利用 hany-plus 技能帮我规划整个项目，包含前端、后端、数据库、部署方案的需求访谈。
项目背景是 [简要描述项目背景]。
请帮我按以下顺序进行系统化访谈：
1. 整体架构与核心域划分
2. 前端交互流程与界面需求
3. 后端服务设计与 API 规范
4. 数据模型与持久化方案
5. 基础设施与 DevOps 流程
每个环节都要输出明确的决策结论和备选方案。
```

### 完整示例

```
我需要为一个 SaaS 产品做完整规划
请帮我按顺序访谈以下模块的需求：
1. 用户认证和权限管理
2. 数据看板
3. 团队协作功能
4. 订阅计费系统

每个模块都请做完整的深度访谈
```

## 4. hany-nice-soft - 代码质量诊断

### 使用场景

- 需要对项目进行全面的代码质量检查
- 代码重构前的评估
- 提交代码前的自查
- 发现潜在的安全问题和性能隐患

### 标准描述

```
帮我扫描当前项目的代码质量
```

### 描述优化

**hany-nice-soft - 完整诊断**

```
使用 hany-nice-soft 技能帮我对当前项目进行全面的代码质量诊断。
请扫描所有代码文件，检查潜在的安全漏洞、性能问题、代码异味和最佳实践违规情况。
按照严重程度（Critical、High、Medium、Low）分类整理发现的问题，并为每个问题提供具体的修复建议。
最终生成一份完整的 Markdown 诊断报告。
```

**hany-nice-soft - 特定范围扫描**

```
利用 hany-nice-soft 技能帮我扫描 src/api/ 目录下的代码文件，重点关注安全性和错误处理。
请检查是否存在 SQL 注入风险、敏感信息泄露、未处理的异常等问题。
扫描结果请保存到 docs/diagnosis.md 文件中。
```

**hany-nice-soft - 提交前自查**

```
使用 hany-nice-soft 技能帮我在提交代码前进行一次快速自查。
请重点检查代码中的 console.log、debugger 语句、TODO/FIXME 注释，以及潜在的空指针风险。
输出结果请直接展示在控制台，不用保存文件。
```

### 完整示例

```
帮我扫描当前项目的代码质量
检查安全漏洞、性能问题和代码异味
按严重程度分类并提供修复建议
```

## 6. brainstorming - 头脑风暴与设计

### 使用场景

- 在开始任何创造性工作之前
- 创建新功能或组件
- 修改现有系统行为
- 需要先理解需求再实施

### 标准描述

```
帮我为 [功能/项目] 进行头脑风暴
```

### 描述优化

**brainstorming - 新功能设计**

```
使用 brainstorming 技能帮我设计一个实时协作编辑功能。
在开始编码前，请先帮我探索项目上下文，了解现有代码库的架构。
然后提出 2-3 种不同的技术方案，分析每种方案的优缺点并给出推荐。
请分节展示设计方案，在每节后征求我的确认意见，最后编写设计文档保存到 docs/specs/ 目录。
```

**brainstorming - 复杂系统重构**

```
利用 brainstorming 技能帮我重构现有的用户认证系统。
请先全面了解当前实现的问题和痛点，然后提出重构方案。
需要考虑向后兼容性、迁移策略、性能影响等因素。
请提供视觉化的架构图对比新旧设计，让我能直观地理解变化。
```

**brainstorming - 可视化辅助设计**

```
使用 brainstorming 技能帮我设计新的仪表盘界面。
我觉得这个功能用视觉化的方式讨论会更有效，请提供浏览器中的可视化辅助工具，让我能看到不同的布局方案和交互效果。
在设计过程中请一次问一个问题，确保我充分理解每个决策点。
```

### 完整示例

```
帮我设计用户通知系统功能
先了解项目上下文，再提出设计方案
分节展示并逐节确认，最后编写设计文档
```

## 7. dispatching-parallel-agents - 并行代理调度

### 使用场景

- 面对多个独立任务可以并行处理
- 测试文件失败且根因不同
- 多个子系统独立出现问题
- 需要高效处理多个独立问题

### 标准描述

```
帮我并行处理这些任务：[任务列表]
```

### 描述优化

**dispatching-parallel-agents - 多测试文件失败**

```
使用 dispatching-parallel-agents 技能帮我处理多个测试文件的失败问题。
当前有 test/auth.test.js、test/checkout.test.js、test/api.test.js 三个文件失败，看起来是不同的根因。
请为每个文件派一个专门的代理去调查和修复，每个代理都要有明确的任务范围、清晰的目标和约束条件。
最后汇总所有代理的修复结果。
```

**dispatching-parallel-agents - 独立子系统问题**

```
利用 dispatching-parallel-agents 技能帮我处理三个独立的子系统问题：
数据库连接池优化、
缓存策略调整、
日志系统升级。
这三个问题互不相关，可以并行处理。
请为每个问题创建专注的代理任务，每个任务都包含具体的目标、限制条件和预期输出格式。
```

**dispatching-parallel-agents - 避免共享状态干扰**

```
使用 dispatching-parallel-agents 技能帮我重构项目中的多个独立模块。
请注意这些模块之间没有共享状态或依赖关系，可以安全地并行处理。
请为每个模块分配一个专门的代理，确保每个代理只关注自己的任务范围，不修改其他模块的代码。
```

### 完整示例

```
帮我并行处理这三个测试文件的失败问题
每个文件派一个专门的代理去调查
最后汇总所有修复结果
```

## 8. executing-plans - 执行计划

### 使用场景

- 有书面的实现计划需要执行
- 需要在独立会话中执行带检查点的任务
- 计划已经写好，现在需要落地实施

### 标准描述

```
帮我执行这个实现计划：[计划文件路径]
```

### 描述优化

**executing-plans - 按步骤执行**

```
使用 executing-plans 技能帮我执行 docs/superpowers/plans/feature-plan.md 中的实现计划。
请先仔细阅读并审查计划，识别任何问题或疑问。
然后创建待办事项列表，逐个执行每个任务。
每个步骤都要严格按照计划进行，并在完成后进行验证。
全部完成后使用 finishing-a-development-branch 技能收尾。
```

**executing-plans - 遇到阻塞时求助**

```
利用 executing-plans 技能帮我执行重构计划。
在执行过程中，如果遇到任何阻塞（缺少依赖、测试失败、指令不清晰等），请立即停下来向我询问，不要猜测或强行推进。
每完成一个里程碑后都要进行验证，确保质量。
```

**executing-plans - 多步骤验证**

```
使用 executing-plans 技能帮我执行这个复杂的部署计划。
计划包含数据库迁移、配置更新、代码部署、回滚方案等多个步骤。
请在每个关键步骤后都进行验证检查，确保一切正常后再继续下一步。
如果任何验证失败，立即暂停并报告问题。
```

### 完整示例

```
帮我执行 docs/superpowers/plans/2025-01-15-feature.md 中的计划
先审查计划，再创建待办事项逐个执行
每个步骤完成后进行验证
```

## 9. finishing-a-development-branch - 完成开发分支

### 使用场景

- 实现完成，所有测试通过
- 需要决定如何集成工作
- 选择合并、PR、保留或丢弃
- 工作区清理

### 标准描述

```
帮我完成这个开发分支的收尾工作
```

### 描述优化

**finishing-a-development-branch - 完整收尾流程**

```
使用 finishing-a-development-branch 技能帮我完成当前开发分支的工作。
首先验证所有测试通过，然后检测当前环境状态：
这是普通仓库还是 Git 工作树？
在分离的 HEAD 上还是在命名分支上？
根据检测结果展示相应的选项菜单，让我选择是合并到主分支、推送创建 PR、保留分支还是丢弃工作。
```

**finishing-a-development-branch - PR 创建**

```
利用 finishing-a-development-branch 技能帮我准备 Pull Request。
在验证测试通过后，请帮我推送当前分支到远程仓库，然后创建一个描述清晰的 PR。
PR 标题应该概括主要变更，正文包含变更摘要和测试计划。
请保留工作区，以便我后续根据评审意见进行修改。
```

**finishing-a-development-branch - 本地合并清理**

```
使用 finishing-a-development-branch 技能帮我把当前功能分支合并到主分支。
请先检出主分支并拉取最新代码，然后合并功能分支，在合并后的代码上再次运行测试验证。
确认一切正常后，删除功能分支，如果是 Git 工作树环境，也要清理相应的工作区目录。
```

### 完整示例

```
帮我完成当前开发分支的收尾
先验证测试，再检测环境，展示选项
按我的选择执行相应的操作
```

```
帮我创建一个高品质的前端界面
```

```
3. REFACTOR：清理代码，保持测试通过

```
