# Hany Plus（循环深度访谈）

## 概述

Hany Plus 是 hany-deep 技能的循环包装器。它会为用户的每个新想法或问题重复调用 hany-deep，在每个访谈周期完成后询问用户是否还有更多问题需要探索。这使得跨多个主题的持续需求收集成为可能，无需每次手动重新触发技能。

## 使用场景

- 用户说 "hany-plus"、"hany+"、"连续访谈"、"循环访谈"、"多个问题"
- 用户有多个模糊的想法需要按顺序探索
- 用户想要循环运行 hany-deep 直到决定停止
- 用户想要批量进行多个需求收集会话

## 执行流程

### 第一步：初始化循环

1. 设置循环计数器：`round = 1`
2. 向用户宣布：

> 正在启动 Hany Plus 循环。我将为你的每个想法/问题运行 hany-deep。每个会话完成后，我会询问你是否还有更多问题需要探索。
>
> **请告诉我你的第一个问题或想法是什么？**

### 第二步：运行 Deep Interview

1. 使用用户的想法调用 hany-deep 技能：
   - 使用 `Skill("hany-deep")` 并将用户输入作为参数
   - 让 hany-deep 执行完整的工作流程（苏格拉底式问答、模糊度评分、规格结晶、执行桥接）
2. 等待 hany-deep 完成整个周期

**⚠️ 关键约束：必须等待 hany-deep 完全结束，包括用户选择的任何执行阶段，才能进入下一步。**

### 第三步：强制询问是否继续（必须执行）

在 hany-deep 完全结束后（包括用户选择的任何执行），**必须**通过 `AskUserQuestion` 提出继续问题。

**⚠️ 关键约束：此步骤是强制性的，不允许跳过，不允许中途退出，必须使用 AskUserQuestion 工具进行提问。**

**必须使用的代码：**
```python
AskUserQuestion(
    questions=[{
        "id": "continue_interview",
        "question": f"访谈会话 {round} 已完成。\n\n**你还有其他问题或想法需要探索吗？**",
        "options": ["是，我还有更多问题", "否，我完成了"],
        "multiSelect": false
    }]
)
```

**禁止的替代方式：**
- 禁止使用普通文本询问
- 禁止使用其他工具询问
- 禁止直接结束而不询问
- 禁止假设用户的选择

### 第四步：处理用户响应

**如果用户选择"是，我还有更多问题"：**

**⚠️ 关键约束：必须再次使用 AskUserQuestion 进行提问，询问下一个想法是什么。禁止使用普通文本询问。**

1. 递增计数器：`round += 1`
2. **必须** 使用 AskUserQuestion 询问下一个想法：

```python
AskUserQuestion(
    questions=[{
        "id": "next_idea",
        "question": "请告诉我你的下一个问题或想法是什么？",
        "options": [],
        "multiSelect": false,
        "freeText": true
    }]
)
```

3. 等待用户提供新想法
4. 返回第二步，使用新的想法继续

**禁止的替代方式：**
- 禁止使用普通文本询问下一个想法
- 禁止直接开始处理而不询问
- 禁止假设用户的想法
- 禁止在用户未提供新想法的情况下继续

**如果用户选择"否，我完成了"：**
1. 宣布："Hany Plus 循环已完成。共完成 {round} 个访谈会话。"
2. 结束技能

## 强制执行规则（必须严格遵守）

### 核心原则：无条件执行 hany-deep

**⚠️ 最重要的规则：无论用户的问题看起来多么简单、多么明确、多么"不需要访谈"，都必须运行完整的 hany-deep 流程。**

AI 不得基于以下任何理由跳过或简化 hany-deep：
- 问题看起来很简单
- 问题看起来很明确
- 用户似乎已经知道想要什么
- 问题只需要几行代码
- 问题看起来是"直接执行"类型
- 任何其他主观判断

**hany-deep 的价值不仅在于澄清模糊需求，还在于：**
- 暴露用户未意识到的假设
- 发现潜在的边界情况
- 确保技术方案的完整性
- 建立可测试的验收标准
- 记录决策过程供未来参考

### 必须执行项（MUST）

- **必须** 为每个用户想法运行完整的 hany-deep 流程 - 无论问题复杂度如何
- **必须** 让 hany-deep 每次都运行完整的工作流程 - 不得跳过或简化任何阶段
- **必须** 在 hany-deep 完全结束后才询问继续问题（包括用户选择的执行阶段）
- **必须** 使用 `AskUserQuestion` 工具进行继续选择 - 不得使用其他方式询问
- **必须** 跟踪循环计数器并在继续提示中显示它
- **必须** 每次循环都强制询问继续问题 - 即使用户看起来已经完成也要询问
- **必须** 等待用户明确选择后才能决定是否继续 - 不得自行判断
- **必须** 在每次 hany-deep 完成后立即执行强制提问 - 不允许中途退出
- **必须** 使用 AskUserQuestion 的标准格式进行提问 - 不得使用普通文本

### 禁止项（MUST NOT）

**关于 hany-deep 执行：**
- **禁止** 基于问题复杂度判断是否运行 hany-deep - 所有问题都必须运行
- **禁止** 认为"这个问题很简单，不需要访谈"而跳过 hany-deep
- **禁止** 认为"用户已经很清楚想要什么"而跳过 hany-deep
- **禁止** 认为"这个问题只需要几行代码"而跳过 hany-deep

**关于强制提问：**
- **禁止** 在 hany-deep 完成后直接退出而不询问继续问题
- **禁止** 在 hany-deep 完成后中途退出而不执行强制提问
- **禁止** 跳过继续问题 - 必须始终强制询问
- **禁止** 使用普通文本询问继续问题 - 必须使用 AskUserQuestion 工具
- **禁止** 使用其他工具替代 AskUserQuestion 进行继续提问

**关于循环控制：**
- **禁止** 自行决定循环结束 - 必须等待用户选择"否"
- **禁止** 在用户选择"是"后不询问下一个想法就继续
- **禁止** 在用户选择"是"后使用普通文本询问下一个想法 - 必须使用 AskUserQuestion
- **禁止** 在用户选择"是"后直接开始处理而不询问下一个想法
- **禁止** 假设用户没有更多问题 - 必须明确询问
- **禁止** 在未收到用户选择前就结束循环
- **禁止** 将多个想法合并到单个 hany-deep 调用中 - 每个想法都有自己的完整周期

**关于用户口头表达（防漏洞）：**
- **禁止** 根据用户说"够了"、"不用了"、"直接做吧"就结束 hany-plus 循环
- **禁止** 根据用户说"就这样吧"、"可以了"、"没什么了"就结束循环
- **禁止** 根据用户说"我没什么问题了"、"没有其他想法了"就结束循环
- **禁止** 根据用户说"你决定吧"、"你看着办"、"随便"就结束循环
- **禁止** 根据用户说"我再想想"、"让我想想"、"稍等"就结束循环
- **禁止** 根据用户说"我没什么要补充的了"、"没有其他要求了"就结束循环
- **禁止** 根据用户说"就这样定了"、"就这个吧"、"就按这个来"就结束循环
- **禁止** 根据用户的满意度、反馈、或任何主观判断跳过提问
- **禁止** 根据对话历史、上下文、或任何主观判断跳过提问

**关于 hany-deep 内部操作：**
- **禁止** 将用户在 hany-deep 中的"够了"理解为结束整个 hany-plus 循环
- **禁止** 将用户在 hany-deep 中的"取消"理解为结束整个 hany-plus 循环
- **禁止** 因为 hany-deep 达到上限就结束 hany-plus 循环
- **禁止** 因为 hany-deep 执行失败就结束 hany-plus 循环

### 违规处理

**如果 AI 试图跳过 hany-deep：**
用户应该明确指出：
- "必须运行 hany-deep"
- "不允许跳过访谈"
- "按规则执行"

**如果 AI 试图中途退出而不强制提问：**
用户应该明确指出：
- "必须强制提问"
- "不允许中途退出"
- "使用 AskUserQuestion 提问"

**如果 AI 使用普通文本询问而不是 AskUserQuestion：**
用户应该明确指出：
- "必须使用 AskUserQuestion"
- "不允许使用普通文本询问"

AI 收到此类提醒后，必须立即停止当前操作，重新运行完整的流程，包括强制使用 AskUserQuestion 进行继续提问。

## 常见问题处理（防退出漏洞补全）

### 漏洞 1：用户在 hany-deep 中说"够了"、"不用了"、"直接做吧"

**场景：** 用户在 hany-deep 的苏格拉底式问答阶段说"够了"、"不用问了"、"直接做吧"、"跳过吧"。

**处理方式：**
- 这是 hany-deep 内部的提前退出，不是 hany-plus 循环的退出
- hany-deep 会处理这个请求（可能会显示警告并允许提前退出）
- **hany-plus 必须继续执行强制提问**，询问用户是否还有其他想法
- **禁止** 将用户在 hany-deep 中的"够了"理解为结束整个 hany-plus 循环

### 漏洞 2：用户说"就这样吧"、"可以了"、"没什么了"

**场景：** 用户在 hany-plus 的继续提问阶段说"就这样吧"、"可以了"、"没什么了"、"不用了"。

**处理方式：**
- **必须** 使用 AskUserQuestion 进行正式提问，不得根据用户的口头表达判断
- 即使用户说"没什么了"，也必须提供选项让用户明确选择
- 用户必须通过点击选项来确认，口头表达不算数
- **禁止** 根据用户的口头表达直接结束循环

### 漏洞 3：hany-deep 执行失败或超时

**场景：** hany-deep 执行过程中出现错误、超时、或异常退出。

**处理方式：**
- 报告错误信息给用户
- **必须** 仍然使用 AskUserQuestion 询问用户是否想要继续探索其他想法
- 不要因为一个想法失败就结束整个循环
- 用户可以选择"是"继续下一个想法，或选择"否"退出

### 漏洞 4：用户说"继续"但没有给新想法

**场景：** 用户选择"是，我还有更多问题"后，说"继续"但没有提供具体想法。

**处理方式：**
- **必须** 明确询问用户："请告诉我你的下一个问题或想法是什么？"
- 不要假设用户没有想法，也不要直接开始处理
- 等待用户提供具体想法后才能继续
- **禁止** 在用户未提供新想法的情况下直接结束或跳过

### 漏洞 5：AI 认为已经完成所有问题

**场景：** AI 根据对话历史判断用户"已经问完了"、"没有更多问题了"。

**处理方式：**
- **绝对禁止** AI 自行判断用户是否还有问题
- 无论对话历史如何，都必须使用 AskUserQuestion 进行正式提问
- 只有用户通过点击选项明确选择"否"才能结束循环
- **禁止** 根据上下文、对话历史、或任何主观判断跳过提问

### 漏洞 6：用户说"我没什么问题了"

**场景：** 用户说"我没什么问题了"、"没有其他想法了"。

**处理方式：**
- **必须** 使用 AskUserQuestion 进行正式提问
- 用户的口头表达不能替代正式的选择
- 提供选项让用户明确选择，口头表达不算数
- **禁止** 根据用户的口头表达直接结束循环

### 漏洞 7：hany-deep 达到硬上限（20轮）

**场景：** hany-deep 达到最大轮次限制（20轮），自动结束。

**处理方式：**
- hany-deep 会自动结束并生成规格
- **hany-plus 必须继续执行强制提问**，询问用户是否还有其他想法
- **禁止** 因为 hany-deep 达到上限就结束 hany-plus 循环

### 漏洞 8：用户在 hany-deep 中说"stop"、"cancel"、"abort"

**场景：** 用户在 hany-deep 中说"stop"、"cancel"、"abort"、"取消"。

**处理方式：**
- 这是 hany-deep 内部的取消操作，不是 hany-plus 循环的退出
- hany-deep 会处理这个请求（保存状态并退出）
- **hany-plus 必须继续执行强制提问**，询问用户是否还有其他想法
- **禁止** 将用户在 hany-deep 中的"取消"理解为结束整个 hany-plus 循环

### 漏洞 9：用户说"下一个"但没有具体内容

**场景：** 用户选择"是"后，说"下一个"但没有提供具体想法。

**处理方式：**

- 不要假设用户没有想法，也不要直接开始处理
- 等待用户提供具体想法后才能继续
- **禁止** 在用户未提供新想法的情况下直接结束或跳过

### 漏洞 10：会话中断或超时

**场景：** 会话被中断（网络问题、超时、系统错误等）。

**处理方式：**
- 如果会话中断，下次启动时应该询问用户是否想要继续之前的循环
- 记录当前循环状态（轮次、已完成的想法）
- 用户可以选择继续或重新开始
- **禁止** 因为会话中断就丢失循环状态

### 漏洞 11：用户说"你决定吧"、"你看着办"

## 错误处理

**如果 hany-deep 执行失败：**
- 报告错误，但仍然询问用户是否想要继续探索其他想法
- 不要因为一个想法失败就结束整个循环

**如果用户在 hany-deep 中取消：**
- 将其视为该想法的完成，然后询问是否继续
- 不要假设用户想要结束所有访谈

**如果会话中断：**
- 下次启动时询问用户是否想要继续之前的循环
- 记录当前循环状态（轮次、已完成的想法）
- 用户可以选择继续或重新开始

## 状态跟踪

在整个循环过程中，跟踪以下信息：
- 当前循环轮次（round）
- 已完成的想法数量
- 每个想法的最终状态（完成/取消/失败）

在每次继续提示中显示这些信息，帮助用户了解进度。
